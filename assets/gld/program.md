# 研究流程說明

本 repo 的目標不是預測 GLD 幾天後的精確價格，而是判斷：

`今天是否是一個適合中期買入的進場點`

目前預設標記方式為：

- 在未來 `60` 個交易日內
- 若先碰到 `+8%`，標記為 `1`
- 若先碰到 `-4%`，標記為 `0`
- 若兩者都沒先明確碰到，或同日順序不明，視為 `neutral`，預設不納入訓練

## 核心原則

1. 先維持標記定義穩定，再做特徵與模型實驗。
2. 只有在標記定義明確不合理時，才調整 horizon 或 barrier。
3. 每次只改一個主要因素，避免無法歸因。
4. 任何實驗，不論有沒有突破，都要留下紀錄。
5. 正式最佳版本只能來自可重跑、可驗證的設定。

## 研究主循環

本 repo 的研究流程不是一次性清單，而是重複循環：

1. 試著完成 `task.md` 裡的任務
2. 根據 `ideas.md` 與本輪結果，進一步拆出新的任務，並更新 `task.md`
3. 回到第 1 步

這是本 repo 最重要的工作方式。

## 標準工作流程

每一輪研究請遵守以下順序：

1. 先查看 `task.md`
   目的：確認本輪正式 backlog，避免重複做已完成的實驗。

2. 挑選並執行 `task.md` 中的任務
   通常會包含：
   - 執行 `prepare.py`
   - 執行 `train.py`
   - 或進行特徵、標記、參數、驗證相關修改
   執行原則：
   - 若 `task.md` 中有多個明確、可直接執行的任務，單輪應盡量一次做完，而不是只做一小部分就提前結束
   - 除非遇到高風險分岔、外部依賴、或需要額外決策，否則不要因為任務量小而等待下一次使用者回應

3. 檢查主要指標
   至少要看：
   - `validation_f1`
   - `validation_accuracy`
   - `validation_bal_acc`
   - `test_f1`
   - `test_accuracy`
   - `test_bal_acc`

4. 判斷是否為有效突破
   預設以 `validation_f1` 為主要指標。
   若 `validation_f1` 提升，但 `validation_bal_acc`、`test_f1` 明顯惡化，不能直接視為正式最佳，需要額外驗證。

5. 將每次正式實驗結果記錄到 `results.tsv`
   不論是否突破，都要留下紀錄。
   `results.tsv` 是完整歷史帳本，用來查詢做過哪些實驗，以及避免重複。

6. 更新 `task.md`
   如果某個任務已完成，請在後方補上實際表現，例如：
   `Performance: validation_f1=0.7123, validation_bal_acc=0.5410, test_f1=0.6891`
   `task.md` 是任務摘要，不需要記錄所有中間實驗，只需保留該任務最具代表性的結果或結論。

7. 回顧本輪結果與 `ideas.md`
   目的：
   - 看哪些方向有效
   - 看哪些方向應停止
   - 看 `ideas.md` 裡哪些想法值得進入下一輪

8. 將值得執行的新想法拆成更小的任務，更新 `task.md`
   原則：
   - 只放下一輪真的要做的內容
   - 不要把所有靈感一次塞進 `task.md`
   - 若需要更細的參數掃描，應先開探索任務，再開細化任務

9. 完成本輪後，整理本輪修改並執行 `git commit` 與 `git push`
   原則：
   - 只提交本輪正式任務直接相關的檔案
   - commit message 要能清楚反映本輪研究主題
   - push 前確認 `results.tsv`、`task.md`、必要程式碼修改都已同步

10. 回到 `task.md`，開始下一輪

簡單說：

- `task.md` 是本輪正式任務
- `ideas.md` 是候選想法池
- `results.tsv` 是歷史結果帳本
- 研究循環就是「做 task -> 看結果 -> 從 ideas 拆新 task -> 再做 task」

## 什麼叫做突破

預設判定方式：

- 首先看 `validation_f1` 是否高於目前最佳
- 若只是非常小的提升，還要一起看：
  - `validation_bal_acc` 是否沒有明顯變差
  - `test_f1` 是否沒有大幅惡化
  - 是否能穩定重跑出相同結果

若只是 validation 小幅變好，但 test 或穩定性明顯變差，應先標記為觀察中，不要急著升級成正式最佳。

## results.tsv 記錄規則

`results.tsv` 欄位固定為：

- `commit_id`
- `validation_f1`
- `validation_accuracy`
- `validation_bal_acc`
- `test_f1`
- `test_accuracy`
- `test_bal_acc`
- `status`
- `description`

其中：

- `status = live`
  代表這個版本是目前正式拿來輸出即時訊號的 live 演算法；同時間原則上只應有一筆 `live`

- `status = candidate`
  代表這個版本值得保留並持續追蹤，但目前還不是正式上線版本

- `status = discard`
  代表這一輪沒有超越現有最佳，或雖然局部變好但不值得升級

## task.md 使用規則

`task.md` 是研究 backlog，不是單純待辦清單。

規則如下：

1. 任務做完就打勾。
2. 任務後面要補 `performance`。
3. 如果任務已執行但結果不好，也算完成，一樣要打勾。
4. 若任務拆成多輪，最後一版要保留最具代表性的結果。
5. 每輪做完後，應依據 `ideas.md` 與本輪結果，重整下一輪的 `task.md`。
6. `task.md` 可以比傳統待辦清單更厚，應容納一整輪足夠的工作量，而不是只放 2 到 3 個小任務。
7. 若某一方向已確認有效，下一輪 backlog 應主動展開該方向的延伸驗證、交易規則比較、與穩定性檢查，不必等使用者逐項催促。

## ideas.md 使用規則

`ideas.md` 是想法池，不是正式任務表。

規則如下：

1. 還沒排進本輪的想法，先放在 `ideas.md`。
2. `ideas.md` 裡的內容可以是方向、問題、候選特徵、候選模型，不必一開始就拆得很細。
3. 若某個想法確定要做，應拆成可執行的小任務後移到 `task.md`。
4. 不要把整個 `ideas.md` 原封不動複製到 `task.md`。
5. 每輪結束後，都要重新檢查 `ideas.md`，挑出值得進入下一輪的內容。

## 什麼情況可以改標記定義

只有在以下情況，才考慮改 target：

1. 標記後資料極度失衡。
2. 大量樣本都落在 neutral，導致有效樣本太少。
3. 目前 barrier 設定與真實交易目標不一致。
4. 研究方向從「中期買點」明確切換成其他任務。

若要改標記定義，例如：

- `60d +8%/-4%` 改成 `60d +10%/-5%`
- `drop-neutral` 改成 `keep-all`

請視為新的研究軸線，必須在 `results.tsv` 和 `task.md` 清楚註記。

## 建議優先研究方向

目前建議優先順序：

1. 驗證目前標記下的 class balance 與 neutral 比例
2. 嘗試更適合中期交易的特徵
3. 增加簡單回測摘要
4. 比較不同 barrier 組合
5. 再考慮更複雜模型

## 注意事項

1. 不要只看單一指標。
2. 不要因為很小的 validation 提升，就忽略 test 退化。
3. 不要把多個大改動混在同一輪。
4. 不要覆寫現有最佳結果而不留下紀錄。
5. 每次升級正式預設前，都要重新驗證一次。
