Performance: `xgboost + future-return-top-bottom-30pct + ret_60 + distance_to_252_high` raised `test_bal_acc` to `0.5298`, the best MU test-side score so far, but `validation_bal_acc=0.4194` stayed far too weak for adoption.

Performance: none are stable enough to adopt yet. `tb20 + ret_60 + vol_ratio_20_120` collapsed to `test_bal_acc=0.5000` with `test_positive_rate=1.0000` in fold 3, while `tb30 + ret_60 + rolling_return_120 + distance_to_252_high` had one fold at `test_positive_rate=0.9958`. The most reasonable compromise is `tb30 + ret_60 + vol_ratio_20_120`: fold 1 and fold 3 both stayed above `0.55` test balanced accuracy, but fold 2 still broke down to `0.4943`.
- [x] Compare operator-style rules on the new MU pass-level candidates. Performance: `tb20 + ret_60 + vol_ratio_20_120` is now the strongest execution-style candidate rather than the safest model reference. Its best static rules were `top_12.5pct` and `top_15pct`, both with `100%` hit rate on the held-out test slice and roughly `44%` average return, while the 4-fold forward check kept `top_15pct` as the winner with `12` trades, `83.33%` hit rate, and `21.62%` average return. By contrast, `tb30 + ret_60 + vol_ratio_20_120` was the more stable model reference, but its best forward operator rule only reached `16` trades, `68.75%` hit rate, and `11.02%` average return on `top_12.5pct`.
- [x] Check whether benchmark-relative context improves the new MU `vol_ratio` lines. Performance: it did not. Adding `QQQ` or `SMH` relative-strength features (`rs_vs_benchmark_60`, `price_ratio_benchmark_z_20`) generally reduced `test_bal_acc` into the `0.49-0.53` zone and reintroduced unstable folds with `test_positive_rate` near `1.0`, so benchmark context is not the missing unlock for MU right now.
- [x] Check whether a simple regime-dual-logistic split stabilizes MU. Performance: it did not. Using `above_200dma_flag`, `distance_to_252_high`, or `slope_60` as the gating feature failed to beat the plain logistic `vol_ratio` leaders, and the stronger-looking variants still broke down in walk-forward.
- [x] Check whether XGBoost improves the MU `vol_ratio` candidates. Performance: it did not. Both `tb20/tb30 + ret_60 + vol_ratio_20_120 + xgboost` variants lifted `headline_score` into the `0.65-0.68` range, but they also pushed `test_positive_rate` back above `0.80` and dropped `test_bal_acc` below `0.50`, so the tree model made MU less usable, not more.
- [x] Tighten the operator sensitivity around the surviving MU `vol_ratio` candidates. Performance: `tb20 + ret_60 + vol_ratio_20_120` remains the best execution path, and its percentile overlays are effectively concentrated around the same cutoff band. `top_12.5pct` and `top_15pct` both kept `3` static trades at `100%` hit rate and roughly `44%` average return, while a simple fixed cutoff at `0.510` reproduced the same clean `3`-trade, `100%` hit-rate profile. `tb30 + ret_60 + vol_ratio_20_120` was less sensitive: `top_10pct` gave the best average return at `29.67%`, while `top_12.5pct` to `top_17.5pct` stayed clustered around `22-23%` with `80%` hit rate.
- [x] Check whether a simple cooldown rule improves the surviving MU overlays. For now, cooldown is an interesting side study, not a preferred operator rule.
- [x] Check whether a confirmation-style dual overlay between the `tb20` execution line and the `tb30` model-reference line improves robustness. Performance: it only helped partially. On the held-out static split, `tb20 top_15pct`, `tb20 fixed_0.510`, and `tb30 top_12.5pct` all converged to the same clean 3-trade profile, so dual confirmation added almost no extra filtering there. In 4-fold forward-style evaluation, the intersection rule did improve the strong fold (`fold_3 avg_return=43.69%` on `3` trades), but it did not fix the weak one (`fold_1 hit_rate=33.33%`, `avg_return=-1.33%`). That means confirmation is useful as an execution tightening tool, not as a true regime-stability solution.

## Notes

- Default label config: `60d +12%/-6% drop-neutral`.
- This asset was scaffolded from the cross-asset first-round batch universe.
- MU is not currently blocked by missing code paths; it is blocked by unstable label/model behavior. The strongest lines so far either stay too positive-rate heavy or become badly validation-test asymmetric.
- MU now has multiple pass-level static candidates, so the next question is no longer "can MU pass the gate at all?" but "which candidate survives walk-forward and operator-rule pressure without collapsing into regime-specific behavior?"
- Current working split:
- `tb30 + ret_60 + vol_ratio_20_120` is the best balance-first model reference.
- `tb20 + ret_60 + vol_ratio_20_120 + top_15pct` is the best execution-style overlay candidate.
- The currently wired live split is slightly more conservative than the pure research winner: execution now runs through `tb30 + ret_60 + vol_ratio_20_120 + top_12.5pct`, while the stronger `tb20` overlay remains a research-side comparison lane until it proves more stable.
- A practical simplification is possible on the execution side: `tb20 + ret_60 + vol_ratio_20_120 + fixed_0.510` behaves almost the same as the current `top_15pct` overlay, so we can treat those two rules as near-equivalents if we later want a simpler live implementation.
- A confirmation variant is also viable for a stricter execution lane: intersecting `tb20 top_15pct` with `tb30 top_12.5pct` slightly improves the best fold but does not solve the bad fold, so it should be treated as an optional tightening overlay rather than the main execution default.
- The currently rejected directions are: benchmark-relative context, simple regime-dual gating, and XGBoost on the `vol_ratio` feature family.
- A fresh caveat now matters: the historical pass rows in `assets/mu/results.tsv` were recorded before the current cached MU data window and do not fully reproduce as-is today. For current-data work, treat them as prior clues rather than hard truth.
- The best current adjustment is not a new feature family; it is threshold control. `MU` now uses `live_threshold_metric=balanced_accuracy` so the live model threshold does not collapse into the same all-positive behavior we saw under F1-first selection.
